🧠 核心概念 先理解这些——AI 的"语言"和"机制"
01
Token
词元 · 最小处理单位
大模型处理信息的最小单位。一个汉字通常对应1~2个Token，一个英文单词对应1个或多个Token。直接影响成本、速度和上下文长度。
02
Prompt
提示词 · 工作指令
给AI的工作说明书。Prompt质量决定输出质量。AI输出质量 ≈ Prompt质量。
03
Context
上下文 · AI的工作记忆
AI当前能够看到的信息，即工作记忆。上下文窗口（Context Window）决定了AI一次能处理的信息上限——超出窗口上限的内容会被AI遗忘。很多时候，给足Context比精修Prompt更关键。
04
LLM
大语言模型
GPT、Claude、Gemini、DeepSeek本质上都是LLM。可以理解为：拥有大量知识但不了解你公司的新员工。
05
Agent
智能体 · 自主执行
能够自主决策并执行多步骤任务的AI。不只是回答问题，而是主动调用工具、搜索信息、完成任务。
06
RAG
检索增强生成
Retrieval-augmented generation。可以理解为先查资料，再回答。
07
MCP
模型上下文协议
Model Context Protocol，链接与扩展的协议层，是连接AI与外部工具的通用接口。让AI能够访问日历、邮件、数据库等外部资源。
08
Skill
技能文件 · 可复用手册
存储工作偏好和流程规范的使用手册（通常下载后为.md格式）。例如，如果从Github中下载了别人分享的skill，你可以上传给AI，让它保存这个skill以便之后调用，减少重复说明。
09
Harness
编排框架 · 管理系统
管理和调度多个Skill、Agent和工具的工程框架。相当于书架+管理员，让复杂AI工作流有序运行。
10
Hallucination
幻觉 · 警惕AI的"自信谎言"
AI有时会自信地编造不存在的事实、引用或数据。这不是AI在有意"撒谎"，而是生成模型基于概率预测的特性。入门第一课：永远核实AI的关键输出，尤其在专业领域。
11
Fine-tuning
微调 · 给AI补"专业课"
在通用模型基础上，用特定领域数据进一步训练，让AI更懂你的业务。与RAG的核心区别：微调是"内化知识"变聪明，RAG是"查阅资料"找答案，两者可互补使用。
12
API
应用程序接口 · AI集成的基础
Application Programming Interface。让不同软件之间互相通信的标准接口。MCP、Agent、各类AI工具都通过API连接外部系统，是理解"AI如何融入现有软件"的基础。
13
Multi-modal
多模态 · AI的"五官"
AI能够同时处理文字、图片、音频、视频等多种信息形式。上传截图让AI分析、用语音对话、生成图片等都是多模态能力。现代AI早已不只是"文字聊天机器人"。
14
Reasoning Model
推理模型 · "深思熟虑"型AI
o1、DeepSeek-R1等新一代模型，回答前会先进行内部推理链思考。类比：普通模型是"快思考"（直觉回答），推理模型是"慢思考"（逐步推演），更适合数学、逻辑、编程等复杂任务。
15
Prompt Engineering
提示词工程 · 系统性设计输入
不只是写一句话，而是系统性地设计提示词的结构、角色设定、约束条件和输出格式。是"用好AI"的核心竞争力，也是目前职场中最实用的AI技能之一。
🔧 配套工具/格式 然后知道这些——AI 的"手脚"和"载体"
01
GitHub
开放协作平台
开放性科学交流平台。用户可以在这里存储、共享代码并与他人协作编写代码，还可以展示或分享您的代码作品；跟踪和管理代码的更改。
02
HTML / md / csv
常见文件格式
HTML是网页格式；md（Markdown）是AI最顺手的文字格式，处理文档前转成md效率更高；csv是比xlsx更干净的表格格式，AI可直接读取。
03
CLI
命令行界面
Command Line Interface。无图形界面，通过文字命令与计算机交互。AI（如Claude Code）可以通过CLI执行代码、管理文件、运行脚本等，自动化程度高、交互反馈快。
04
IDE
集成开发环境
Integrated Development Environment。专为编程设计的工具，如VS Code、Cursor、PyCharm等。提供代码编辑、调试、版本控制等功能。AI可直接在IDE中与开发者协作编写代码。
05
PowerShell
Windows脚本语言
Windows系统上的高级脚本语言，比CMD命令行功能更强大。可用于系统管理、自动化任务、数据处理等。AI可以帮助编写PowerShell脚本来自动化重复工作。
06
Automation
自动化 · 流程自动化
通过脚本、Agent或工作流让重复性任务自动执行，减少人工干预。AI可以帮助设计自动化流程：如定期生成报表、自动发送邮件、数据定期更新等，提高工作效率。

5层递进逻辑框架

第1层：大脑 / LLM
对应概念：LLM/大语言模型
代表对象：GPT、Claude、Gemini
关键词：Token、Context Window、多模态、推理模型
核心：基础能力
这一层是地基，决定 AI 的基础能力：
输入的Token的多少和质量一定程度上决定了成本以及输出的速度；
上下文窗口的大小则决定了这个AI的工作记忆上限；
多模态则是指AI不止能读文字，还可以读取图片或者听声音；
不同的推理模型，他们的推理能力也是不一样的。
第2层：沟通方式
对应概念：GUI、CLI、桌面版、Prompt Engineering
核心：交互方式
决定人和AI"怎么对话"：GUI（图形界面）和CLI（命令行）是主要的两种方式，而Prompt Engineering（提示词工程）则是“怎么和AI好好的沟通”，决定了AI能否准确理解你的意图并执行你的命令。
第3层：工具和手册
对应概念：RAG、MCP、Skills、Harness
核心：扩展功能
解决AI"知识不新、手不够长"的问题：RAG让AI能查外部资料；MCP是万能接口，连接各种工具和数据源；Skills是预设的操作手册；Harness是调度这一切的运行框架。
第4层：智能体 / Agent
对应概念：Agent、自主决策
核心：自主执行
核心公式是Agent = LLM + 工具 + 自主循环，因此这一层强调的是AI自主决策和执行能力，代表的对象有Claude Code, Codex, Hermes等。
第5层：办公应用
对应概念：IDE、办公环境、自然语言
代表对象：VS Code、Cursor
核心：实际应用
这一层主要指IDE，可以理解为AI工作的"办公环境"；像Cursor、VS Code这类工具，可以理解为办公室，而不同的AI agent是员工。你作为老板可以把不同的AI放到这个办公室里，方便调用。Vibe Coding则是指自然语言提需求，AI负责生成、修改、调试代码，人更多是把关方向、验收结果。

递进关系：基础 → 交互 → 扩展 → 自主 → 应用｜每一层都是下一层的基础，循序渐进学习

💬 Prompt — 如何写更清晰的指令

为什么 AI 回答不好？很多时候不是 AI 不行，而是指令不够清晰。一个好的 Prompt 能让 AI 输出质量提升很多。

📋 Prompt 的标准结构

一个有效的 Prompt 应该包含以下几个要素（不一定全部都用，视情况而定，可以后续慢慢补足）：

[角色] 你是一个专业的HR招聘经理
[任务] 帮我筛选和评估候选人
[背景/上下文] 我们公司是一个50人的SaaS创业公司，招聘的是销售经理
[具体要求]
- 必须有3年以上B2B销售经验
- 需要评估其目标管理能力和团队管理潜力
- 从候选人简历中提取：关键成就、行业经验、管理背景
[输出格式] 用表格展示，包含：匹配度评分、优势、关注点、面试建议
[限制/注意] 不要做主观判断，基于简历中明确信息进行评估

🎯 5个关键的 Prompt 写法技巧

1️⃣ 明确角色

告诉AI你希望它扮演什么角色，会让它的回答更符合专业标准。

❌ 不够清晰
"帮我写招聘文案"
✅ 更好
"你是一个有15年招聘经验的HR总监，帮我写一份吸引高级产品经理的招聘文案"
2️⃣ 提供充足的上下文

AI对你的公司背景了解越多，建议就越贴切。不要假设AI知道你的情况。

❌ 信息不足
"分析一下这个候选人怎么样？" + [简历内容]
✅ 更好
"我们招聘的是B2B SaaS销售，团队规模5人，主要面向国内Fortune 500企业。请分析这个候选人的适配度，重点考察：行业经验、大客户销售能力、工作稳定性。" + [简历内容]
3️⃣ 明确输出格式

告诉AI你想要什么格式的答案（表格、列表、段落、JSON等），会减少后期调整的时间。

❌ 模糊
"列出这个城市的薪酬数据"
✅ 更好
"用表格展示北京地区互联网行业HR的薪酬水平（按职级分类），包含：P5/P6/P7 三个级别，每个级别显示：平均年薪、薪酬范围、主要福利"
4️⃣ 给予具体例子

一个清晰的示例比长篇大论更有效。AI会模仿你的例子来调整回答。

例子：
"我想让AI按照我公司的风格写邮件。我的风格是：简洁、友好、行动导向。

我的邮件示例：
'Hi 小王，周五有个重要会议，涉及你的项目。下午3点会议室2楼见。如果有问题提前告诉我。'

请按照这种风格，帮我写一封关于[主题]的邮件"
5️⃣ 设置限制和边界

告诉AI什么不要做，会避免很多不必要的输出。

❌ 容易出错
"给我一份绩效评估流程"
✅ 更好
"根据我们的企业规模（500人）和风格（偏flat），设计一份绩效评估流程。要求：（1）不超过3层评估，（2）避免过度复杂，（3）重点考察个人目标与公司战略对齐，（4）留出反馈环节但不要超过2周"

💡 HR 工作中的 Prompt 示例

示例1：简历初筛
你是一个有10年招聘经验的HR专家。帮我筛选以下简历，岗位要求如下：

岗位信息：
- 职位：人力资源经理
- 行业：SaaS创业公司
- 必需：3年HR工作经验，了解创业公司运营
- 加分项：有招聘、薪酬设计、员工关系经验

输出格式：
用简单的表格，包含：候选人名字｜经验匹配度(1-5分)｜主要优势｜关注点｜推荐指数

注意：只基于简历内容评估，不做过度推断。
示例2：薪酬分析
你是一个薪酬顾问。帮我分析以下问题：

背景：
- 公司规模：200人的B2B SaaS
- 所在地：上海
- 我的职位：HR总监，工作5年，带过一个3人团队
- 当前薪酬：45万年薪 + 15万年终奖

问题：
1. 这个薪酬在行业中处于什么水平？
2. 根据我的经验和岗位责任，合理的薪酬范围是多少？
3. 我应该关注哪些谈判重点？

输出：简洁的分析，每点3-4句话。
示例3：招聘文案
你是一个资深的业务经理或者HR。帮我写一份招聘文案。

岗位：产品经理
公司背景：AI创业公司，融了B轮，30人规模，产品是一个面向B端的AI工具
工作内容：从0到1建立产品的HR应用场景
要求：
- 风格：简洁、有吸引力、能吸引对AI感兴趣的PM
- 长度：3-4段，每段不超过80字
- 重点强调：创业机会、技术前沿、自主性，而不是稳定性
- 不要用套话，避免"广阔前景、快速成长"之类的虚词

⚠️ 常见的 Prompt 错误

❌ 错误1：期望AI猜测你的隐含需求
不要假设AI知道"我们公司的文化是什么""什么样的候选人符合我们的价值观"。你需要明确说明。
❌ 错误2：信息太多，没有重点
给AI100行背景信息，它可能被淹没在细节中。最好是：核心信息 + 具体例子 + 明确指示。
❌ 错误3：要求AI替代你做决策
不好："告诉我应该雇这个人吗？" 好的："列出这个候选人的5个优势和3个风险点，帮我做决策。"
❌ 错误4：一个Prompt试图解决多个完全不同的问题
不好："帮我优化招聘流程、写招聘文案、设计面试问题" 好的：把这三个分成三次对话，或者在一次对话中明确分段。
❌ 错误5：Prompt太长而没有逻辑分层
用【背景】【任务】【要求】【输出格式】这样的标题来分段，会让AI更容易理解。

✨ Prompt 写作的黄金法则

🎯 第一次问不好，就迭代。 如果AI的回答不符合预期，不是AI的问题，通常是你的Prompt不够清晰。试着：

  • 添加更多背景信息
  • 提供一个你想要的示例
  • 明确说明AI遗漏了什么
  • 改变输出格式
  • 提出后续问题补充细节

与AI的对话是迭代过程，第二、三轮通常比第一轮好得多。

🎯 Skill — 如何构建和复用能力

什么是 Skill？

Skill 是一份给AI的"使用手册"，记录你的工作偏好、标准流程、评估框架等。 当你上传 Skill 给AI时，AI会按照你定义的方式来完成任务。

简单来说：Skill = 你的工作风格 + 常用流程 + 决策标准，打包成一份文档。

📌 为什么要用 Skill？

问题： 每次用AI都要重新解释"我们公司的风格是什么""候选人该怎么评估"
解决方案： 创建一个Skill，AI会记住这些标准，下次可直接调用。
节省时间： 花一定时间在创建你自己的Skill上，后续使用会节省很多时间在重复Prompt上面。
团队一致性： 分享Skill.md给同事，大家的AI都可以用统一的评估标准、风格、流程。
知识积累： Skill就是你工作知识的文档化。你想起来的最佳实践、踩过的坑，都能记录下来。

🔧 Skill 的常见类型

📝 写作风格 Skill

定义你的邮件风格、文案风格、会议纪要格式等。例如："我的邮件风格是简洁、友好、行动导向，避免冗长的开场"

⚖️ 评估标准 Skill

定义候选人如何评估、绩效如何评分、项目如何打分等。例如："候选人评估的3个维度是：经验匹配度、学习能力、文化契合度"

📋 流程模板 Skill

定义标准的工作流程。例如："我们的招聘流程是：简历筛选→初试→复试→终试→offer→入职"

📊 数据标准 Skill

定义数据口径、指标定义。例如："离职率 = 本月离职人数 / 上月末人数"

🎨 公司信息 Skill

定义你的公司背景，让AI每次对话都记得。例如："我们是一个50人的SaaS创业公司，行业是...，融资状态是..."

🚫 禁忌和避免 Skill

告诉AI什么不要做。例如："不要使用专业术语和套话，避免性别歧视的表述"

📚 Skill 的结构和示例

💡 灵活创建原则： 很多时候，你只有模糊的思路，不一定要一开始就创建完整的Skill。可以从最小化版本开始（只有背景+目标+核心标准），然后在使用过程中逐步完善调整。不必追求完美，先用起来，边用边改！

以下是几个不同场景的 Skill 示例，从完整版到简化版都有：

📋 示例1：候选人评估标准（完整版）

# 候选人评估标准 - 销售岗位 ## 背景 我们是一个50人的B2B SaaS创业公司，主要面向传统行业提供HR数字化解决方案。我们需要招聘有互联网背景的销售经理。 ## 目标 帮助我快速评估候选人是否适合我们的销售岗位，特别是考察大客户销售能力和创业精神。 ## 核心评估维度 1. 行业经验 — 必须有互联网/SaaS行业背景，至少3年 2. 销售能力 — 关注：大客户销售、续费能力、团队协作 3. 文化匹配 — 是否认可创业精神、学习意愿、主人翁意识 ## 评分标准 - 5分：完全符合，2个维度以上突出 - 4分：主要符合，1个维度突出，1个维度中等 - 3分：基本符合，各维度均衡 - 2分：有缺陷，1个维度不足 - 1分：不符合，多个维度严重不足 ## 输出格式 用表格展示：维度 | 候选人情况 | 评分 最后补充：总体建议和是否推进面试 ## 禁忌 - 不要过度推断简历外的信息 - 避免"潜力大""发展空间"等虚词 - 避免性别、年龄、家庭背景等歧视内容

🚀 创建 Skill 的两种方法

方法1️⃣：自己写 Skill（自主型）

  1. 第一步：整理思路 — 自己先想清楚这个 Skill 要做什么、标准是什么
  2. 第二步：用 Markdown 写下来 — 按照标准结构写：背景、目标、维度、评分标准、输出格式、禁忌
  3. 第三步：保存和测试 — 保存成 .md 文件，上传到AI并试验一次，看效果
  4. 第四步：迭代改进 — 根据 AI 的输出调整 Skill 内容

适合：有coding基础，并有明确想法、时间充足的人

方法2️⃣：和 AI 沟通来创建 Skill（协作型）

核心思路：Plan → Refine → Finalize
不要一开始直接让AI生成Skill。而是先和AI讨论、确认方向、打磨细节，最后才生成最终版本。

具体步骤：

  1. 第一步：描述需求
    告诉AI你想创建什么Skill。例如："我想创建一个候选人评估的Skill，重点关注销售经验和学习能力"
  2. 第二步：讨论框架
    让AI提出建议的框架。例如AI可能会说："我建议用3个维度来评估，分别是...，你觉得这样可以吗？还是需要调整？"
    这一步很关键：你要确认方向，而不是盲目接受AI的第一版
  3. 第三步：逐项细化
    一个维度一个维度地讨论。"第一个维度是行业经验，我觉得还需要考虑...，你怎么看？"
    AI会根据你的反馈调整。这个过程可能需要3-5轮讨论。
  4. 第四步：确认输出方式
    输出方式可以是指定语言的文字形式，也可以是表格，excel，html，评分表+总结等形式。这些都主要取决于你最终要得到什么。不要让AI自己决定，要你来驱动。例如，如果是文字形式，你希望是什么语言？要给出几种答案？简介的还是详细的？如果是评分标准，你可以和AI讨论并确认5分应该是什么样的？4分呢？或者及格是多少？当你不确定怎么确认这些的时候，你可以询问AI的建议做参考，例如“我不是很确定如何确认评分细节，你能给我一些方向吗？”
  5. 第五步：最终生成
    当你觉得框架、流程和输出形式都没什么问题，你不用着急生成，可以先让AI陪你预演一遍（例如："基于我们刚才讨论的内容，你先不用着急生成skill，先和我过一遍流程"），查漏补缺，再让AI生成最终的Skill。Skill可以直接保存在账号上，下次使用时可以直接调用（例如Claude和Codex CLI是可以直接在对话框里输入斜杠“/”来调用。你也可以下载Skill，安装到别的AI中，前提是这个AI也支持使用Skill。

适合：没有coding基础，想法还不够清晰、希望AI帮助启发、时间灵活的情况

💡 关于 Token 成本的建议：
如果担心和Claude沟通创建Skill会消耗较多Token，可以先用免费工具（如DeepSeek、ChatGPT免费版）来讨论和打磨思路，确认框架和细节都没问题了，再用Claude最后生成定稿版本的Skill。这样既能充分迭代，又能控制成本。

📊 Project — 如何让 AI 持续跟进任务

有时候一个任务需要多次迭代：招聘需要持续跟进候选人进度、培训方案需要多次调整、数据分析需要补充新数据。Project 就是让 AI 记住这个任务的所有上下文，不用每次都从零开始。

🎯 Project 的核心特点

✦ 持久化工作空间： 跨对话保留记忆。你上传的文件、自定义指令、对话历史都会被保存。每次进入 Project，AI 都能记住之前的所有内容。

✦ 自动化读取资源： 可以上传 Skill 文件、校准简历、岗位配置等。Claude 每次进入都会自动读取这些资源，不用每次都重新上传。

✦ 多人协作： Pro/Team 用户可用。Team 版支持多人协作和权限管理，整个团队可以在同一个 Project 中协作。

🤔 Project vs Skill vs Prompt

Prompt 一次性的指令 用于临时任务，不需要记忆
Skill 可复用的标准 用于重复出现的工作（每次招聘都用同一套标准）
Project 长期任务的工作空间 用于一个需要持续推进的具体项目（这个月的校招、这个季度的薪酬调整）

📌 Project 的适用场景

招聘进度跟进
你在跟进3个候选人，需要AI记住每个人的进度、反馈、下一步。而不是每次都重新介绍这3个人的信息。
方案设计和迭代
在 Project 中设计绩效制度，第一版AI给出初稿，你反馈说"这一条太复杂了"，AI在 Project 中直接修改第一版，保持前后一致。
数据分析持续更新
你需要每月生成HR数据报告。建一个 Project，AI每月在同一个 Project 中更新数据，保持历史记录和对比。
团队培训计划
设计一个为期3个月的培训方案。在 Project 中，AI帮你逐周优化，还能记住前面的周、员工的反馈、学习进度等。
员工手册编写
编写公司的员工手册。第一周写第1-3章，AI记住了你的风格；第二周写第4-6章，AI会自动保持一致的风格和深度。

🛠️ 如何在 Claude 中创建 Project

  1. 在 Claude.ai 上，点击左边栏的 "Create Project"
  2. 给 Project 起一个名字，例如"2024年校招进度"
  3. 上传相关的文件（JD、候选人信息、公司背景等）或直接粘贴文本
  4. 在 Project 中开始对话。AI 会把这个 Project 的所有信息作为背景
  5. 每次新对话时，Project 内的信息都会被 AI 记住
  6. 可以为 Project 添加备注、标签，方便后续查找

💡 Project 的最佳实践

✅ 在 Project 中保持清晰的上下文

  • 第一次对话时，清楚地说明这个 Project 的目标、背景、现状
  • 定期总结进度：已完成、进行中、待处理
  • 如果有重要的改变（如需求变化），明确告诉 AI

✅ 利用 Project 进行版本管理

  • 如果需要多个版本（如A方案和B方案），可以在 Project 中并行讨论
  • AI会记住各个版本的细节，帮你对比和选择

✅ 定期导出和备份

  • Project 完成后，导出为 PDF 或 Word，保存到本地或云盘
  • 这既是备份，也是一份最终的工作记录

⚠️ Project 的限制

上下文有限制： Project 中的对话如果太长（通常在50-100条消息后），AI 可能开始遗忘最早的部分。应对办法是定期总结和导出。

不是实时数据库： Project 中的数据不会实时同步到其他系统。如果你需要 HR 系统中的最新数据，需要手动更新给 AI。

💾 Artifact — 如何沉淀 AI 产出

Artifact 是 AI 生成的可复用文件或内容。与 Prompt 的一次性输出不同，Artifact 是被保存、编辑、分享、反复使用的成果。

🎯 Artifact 的核心特点

✦ 单次对话的独立输出物： 是 Claude 在对话中生成的独立内容（代码、工具、卡片、文档等）。

✦ 右侧面板展示： 出现在右侧面板，可以预览、编辑、分享。

✦ 跨对话持久化存储： 不跨对话保存，但可以带持久化存储（Pro 及以上账户）。意味着你可以长期保存和复用 Artifact。

📋 什么是 Artifact？

当你让 AI 生成一个重要的、可以被反复使用的内容时，AI 会在专门的窗口中生成，你可以：

  • 💾 保存： 下载为 Word、PDF、HTML 等格式
  • ✏️ 编辑： 直接在 Artifact 中修改内容
  • 🔄 反复使用： 明年招聘时，直接用去年的招聘文案 Artifact 改一改
  • 🤝 分享： 复制链接分享给同事
  • 📊 版本管理： 同一个 Artifact 可以有多个版本

🎯 Artifact 的常见类型

类型 例子 使用频率
📄 文档 员工手册、招聘指南、OKR模板 经常复用
📝 文案 招聘文案、内部邮件模板、新闻稿 定期修改后复用
📊 表格/流程图 薪酬表、面试流程图、组织结构 频繁更新和参考
🔧 工具/计算器 薪酬计算器、年假计算工具 长期使用
💻 代码 HR系统、数据处理脚本 技术性应用

🛠️ 如何创建和管理 Artifact

1️⃣ 自动生成

当你要求 AI 生成比较长的内容（如一份完整的文档），AI 会自动在 Artifact 窗口中生成。你会看到一个单独的面板。

2️⃣ 编辑和优化

在 Artifact 中你可以直接修改内容，也可以告诉 AI "把第3段的语气改得更友好"，AI 会在 Artifact 中实时更新。

3️⃣ 保存和分享

右上角有"下载""复制链接"等按钮。你可以下载为 .docx 或 .pdf，或者复制链接分享给同事（他们可以看到这个 Artifact，但不能编辑）。

💡 Artifact 的最佳实践

建立 Artifact 库
在你的电脑上建一个文件夹（或在云盘、GitHub），专门保存重要的 Artifact。例如：
📁 HR_Artifacts/
├─ 招聘/
│ ├─ 招聘文案_2024.docx
│ ├─ 面试流程.xlsx
├─ 员工管理/
│ ├─ 员工手册_v2.pdf
│ ├─ 离职流程.docx
版本管理
重要的 Artifact 要做版本管理。例如"员工手册_v1_2024年" "员工手册_v2_修订后"，这样可以追踪变化历史。
定期更新
一些 Artifact 需要定期更新（如招聘指南、薪酬标准）。设一个提醒，每个季度或每年更新一次。
团队共享
把重要的 Artifact 分享给团队。你可以直接分享链接，或者上传到团队的 Wiki/Notion，让大家都能访问和参考。

⚠️ Artifact 的注意事项

隐私和敏感信息： 不要在 Artifact 中放入真实的员工姓名、工资数字、身份证号等敏感信息。用"员工A""薪酬范围"这样的虚指代替。

链接分享的权限： 分享 Artifact 链接时，确认你的 Claude 账户的隐私设置。如果不想让陌生人看到，不要分享链接到公网。

⚙️ 个性化配置 — 让 AI 更懂你

在开始使用 AI 之前，花一点时间配置好你的"Agent 准则"，可以让 AI 更理解你的风格、需求和工作习惯。这样每次对话都会更高效，少做很多重复解释。

1️⃣ 设置你的 Agent 准则

首次使用 AI 时，建议先定义你的"角色画像"和"使用习惯"。

🎭 角色画像包括：
• 你的职位和主要职责（如"我是一位招聘专员，负责校招和中层社招招聘"）
• 你所在的行业和公司规模（如"互联网公司，5000人规模"）
• 你的工作重点（如"重视候选人的学习能力和文化匹配度"）
⚡ 使用习惯包括：
• 你的沟通风格（如"请用简洁语言，避免冗长段落"）
• 你的输出偏好（如"我更喜欢表格而不是段落"、"请添加具体例子"）
• 你的工作节奏（如"我通常在周一和周五做总结"）
• 你关心的指标（如"重点关注成本和效率"）
💡 写准则的技巧： 可以让 DeepSeek、ChatGPT 等免费工具帮你写好初稿，然后再自己调整。完成后，保存为 Skill 或 Project 的开场白，这样每次都能自动调用。

2️⃣ 寻找和复用现成的 Skill

不用所有东西都自己写。很多高手已经配置好了 Skill，可以直接复用。

⚙️ Token 优化 — 如何高效使用

Token 是 AI 理解的"文字单位"。 你输入的每个字、每个空格、AI 生成的每个字，都会消耗 Token。如果你的 Prompt 很长，或者上传了很多文件，会消耗更多 Token。了解如何优化 Token 使用，可以帮你节省成本。

📊 什么是 Token？

  • 大约 1 个中文字 ≈ 1-2 个 Token
  • 1 个英文单词 ≈ 1 个 Token
  • 1 个标点符号 ≈ 1 个 Token
  • 空格和换行也会消耗 Token

例子：一份 1000 字的文档 ≈ 1000-2000 Token

💰 Token 成本和计费

不同的模型，Token 价格不同。一般来说，输入的 Token 价格较低，输出的 Token 价格较高。

举个例子： 如果你的 Prompt 输入了 10,000 Token，AI 输出了 5,000 Token，费用可能是：10,000 × 输入价格 + 5,000 × 输出价格。

🎯 一些节省 Token 的技巧

1️⃣ 精简背景信息

不好："我们公司是一个位于上海的、成立于2018年的、专注于AI驱动的HR解决方案的创业公司，融了B轮2000万美元…"
好的："50人的AI SaaS创业公司，融B轮"

2️⃣ 避免重复提供信息

如果在 Project 或 Skill 中已经说过，不要在 Prompt 中再说一遍。直接说"根据我的招聘标准 Skill，评估这个候选人"。

3️⃣ 只上传必要的文件

要分析薪酬数据，就上传薪酬表。不要把整个企业云盘都上传给 AI。引用文件时可以直接@文件名指定路径，而不是让agent帮你搜。

4️⃣ 用简短的变量名

不好：使用完整的员工名字。好的：用"员工A、员工B"，或者用ID（员001、员002）。

5️⃣ 分解大任务

不好：一个超长的 Prompt 做5件事。好的：分成5个短对话，每个只需要 200-300 Token。

✦ /plan 指令主要适用于 CLI。 复杂内容先让模型把方案定清楚，确定方向对了在执行。在 Claude.ai 网页版中，可以直接告诉 AI "先为我制定一个计划"或"先分析一下思路"。在 Codex 中，有两个相关功能：

• Plan Mode： 开关选项，开启后 AI 会先生成计划再执行。
• 追求模式（Pursuit Mode）： 另一种执行模式，AI 会更深入地追求目标，投入更多的思考和迭代，适合需要高质量结果的复杂任务。

这两个模式都能帮助你减少错误和返工。直接让 LLM 执行复杂任务可能会造成错误或返工，每一次错误尝试都会导致上下文堆积。

6️⃣ 利用流式输出

在 Claude.ai 上，你只需要为最终生成的内容付费。如果你停止了对话，剩余的输出不会被计费。

7️⃣ 使用缓存

如果你通过 API 调用 Claude，可以使用"缓存"功能。第二次使用同样的背景文件或 Skill，费用会降低90%。

另外，某些模型可以将你的 Prompt 临时缓存一定时间（一般是分钟级别），在这个时间内调用 Prompt，所用 token 额度会大大减少（成本1/10左右）。但如果半个小时之后再来发，额度会增大。

8️⃣ 优化输出格式

不好："用长篇幅的段落讲解"。好的："用表格、列表、要点总结"。后者通常会生成更少的 Token。

9️⃣ 批量处理相似任务

一次处理10个相似的简历，比逐个处理节省 Token。AI 可以记住格式，后续输出会更简洁。

🔟 设置预期输出长度

在 Prompt 中明确说"控制在300字以内"或"简洁回答，不超过5个要点"，AI 会生成更短的输出。

1️⃣1️⃣ 避免上下文冗余

多个问题合并成一条消息，不要想到一条指令是一条。出错请纠正，去重新编辑原本的消息，把正确的要求补进去。

✦ /btw 指令主要适用于 CLI。 在 Claude.ai 网页版中，可以直接和 AI 说"这是个临时提问，不需要计入之前的对话"。在 Codex 中，可以开启侧边聊天（Side Panel Chat）功能来替代 /btw，这样侧边的临时对话就不会混入主对话的上下文中。

1️⃣2️⃣ 主动清理上下文

✦ 注：以下指令主要适用于 CLI，如 Claude Code、命令行工具等。

/clear 或 /new： 新任务开新的 session，可以开启全新的上下文。在 Claude.ai 网页版中，可以直接点击"新对话"或告诉 AI "开启新的对话"。

/statusline： 输入这个指令显示上下文使用百分比。在网页版中，你可以直接问 AI "我现在上下文用了多少"。

/compact： 主动压缩上下文。在 Claude.ai 网页版中，可以直接说"帮我压缩一下上下文"或"整理一下对话记录"。一些AI会主动压缩上下文，但一般只会在占95%的时候触发。某些模型还会表现出"上下文焦虑"，即当它们自己接近上下文容量上限时，会过早地开始收尾工作。所以当看到上下文已经超过70%或者感觉运行卡顿时，主动请求压缩或者开新的对话，可以说"压缩上下文时保留工具调用的细节"。

如果要长时间离开，先压缩并指定要保留的内容。

1️⃣3️⃣ 新任务先盯着 Claude

如果发现方向错了，及时止损。按 Esc 按键暂停，然后再重新编辑上一条消息的提示词。

通用小建议： 优先选择编辑而非重写整个文件。除非文件被编辑过，否则不要重复阅读已读过的文件。输出追求简洁，但推理过程必须详尽。

📈 什么时候值得花更多 Token

高价值的任务： 如果这个任务能为公司带来价值（如招聘文案会被反复使用、数据分析会影响决策），值得花更多 Token 做高质量的输出。

重复使用的文件： 如果一个 Artifact 会被使用10次以上（如招聘指南、员工手册），首次多花 Token 打磨质量，后续会节省大量时间和成本。

复杂分析： 有时多花 Token 提供更多背景信息，能让 AI 做出更准确的分析，避免后续的反复修改。

💡 Token 成本 vs 时间成本

最重要的是平衡：不要为了省几块钱的 Token 费用，浪费自己的时间。 如果优化 Token 会让工作变得复杂、效率变低，就不值得。AI 的成本很低，你的时间成本更高。

🚨 常见误区 — 避免踩坑

很多人在使用 AI 时，因为不了解 AI 的特性和局限，踩了很多坑。这里列出最常见的 13 个误区，帮你避免重复犯错。

❌ 误区1：期望 AI 100% 准确
很多人会说："AI 告诉我这个薪酬数据，我直接信了。"结果发现数据有误。

问题： AI 会"幻觉"（编造听起来合理但不真实的信息），特别是涉及具体数字时。AI 会自信地生成看似专业、实则不存在的内容——虚构的薪酬数据、编造的法律条款、杜撰的案例等。

🚨 HR场景中幻觉的危害：
• 薪酬数据错误：AI 说的市场薪酬可能跟实际有偏差，导致招聘困难
• 法律建议错误：AI 可能编造不存在的法律条款，导致公司违规
• 政策错误：AI 可能给出不合规的建议（如社保、劳动法相关）
✅ 解决方案：永远把 AI 的输出作为初稿或参考，需要验证的数据（如薪酬、人数）必须自己查证。最佳实践是"AI 加速，人工验证"。涉及数字、法规、政策的内容，要从官方渠道交叉验证。在 Prompt 中明确说"如果你不确定，请说'我不知道'，而不是猜测"，也能有效减少幻觉。
❌ 误区2：直接复制 AI 生成的文案发给员工
有些人会让 AI 写一份离职邮件，然后直接改个名字就发了。

问题： AI 可能会用模板化的语言，显得不够真诚；可能不了解你的员工的具体情况，显得生硬；最重要的是，这样的沟通缺乏人的温度。
✅ 解决方案：把 AI 的输出当作初稿，一定要自己修改、加入个人化的内容、体现你的关怀。尤其是敏感的人事沟通（离职、降薪、绩效不达预期），不能完全依赖 AI。
❌ 误区3：把敏感信息上传给 AI
有人会把真实的员工名字、工资、身份证号、绩效评分上传给 AI 分析。

问题： 虽然 Claude 等工具有隐私保护，但没有 100% 的保障。一旦信息泄露，影响很大。
✅ 解决方案：使用虚指代（员工A、员工B，或用员工ID）。例如分析薪酬时，用"P级别，3年经验，市场价25-35k"代替"张三，25.5k"。
❌ 误区4：让 AI 替代你做决策
"AI，告诉我该雇这个候选人吗？" "这个员工应该绩效等级是几分？"

问题： AI 可以分析信息，但它无法承担决策的责任。如果因为 AI 的建议导致错误的人事决定，你承担后果，不是 AI。
✅ 解决方案：AI 的角色是"参谋"，不是"决策者"。问 AI "列出这个候选人的5个优势和3个风险"，而不是"我该雇他吗？"。最终决策一定是你来做。
❌ 误区5：每次都从零开始问 AI
第一个月 AI 帮你筛选候选人，每次都要重新解释"我们的评估标准是什么"。

问题： 浪费时间，而且 AI 每次的理解可能略有不同，导致评估不一致。
✅ 解决方案：建立 Skill，记录你的评估标准。第二次直接说"用我的招聘评估 Skill 来筛选"。可以节省大量重复 Prompt 的时间。
❌ 误区6：Prompt 写得太模糊或信息不足
"帮我分析这个候选人" + [一份简历]，啥背景都没有。

问题： AI 不知道你的公司什么样、这个职位什么要求、你关心什么，只能给出很泛泛的回答。
✅ 解决方案：给 AI 充足的背景。"我们的公司是...，这个岗位是...，重点考察...，你看这个候选人如何？"
❌ 误区7：不迭代，一次就放弃
"AI 给的方案不好"，然后就不用了。

问题： 第一次 Prompt 的质量往往不是最好的。大多数情况需要迭代：反馈给 AI，AI 改进，再反馈，再改进。
✅ 解决方案：把与 AI 的交互当作一个迭代过程。"这个方向对，但第3点太复杂，简化一下。" AI 会记住你的反馈，不断改进。通常第2-3轮就能出来不错的结果。
❌ 误区8：混淆 Project 和 Artifact 的用途
有人会把"设计一份招聘指南"放在 Project 中反复迭代，最后生成的内容还留在 Project 里；或者把一份需要持续跟进的任务（如招聘进度）当成 Artifact 保存。

问题： 用错工具，导致工作流程混乱。
✅ 解决方案：明确区分用途

📊 Project 的作用：
• 长期任务的工作空间，AI 保留上下文
• 需要多次迭代和反馈
• 任务进行中，内容还在变化
• 例：招聘一个岗位（涉及多个候选人、多轮反馈）、薪酬方案设计（v1→反馈→v2→反馈→最终版）

💾 Artifact 的作用：
• 最终产出物，可复用的文件
• 需要保存、下载、分享
• 内容已经定型或成熟
• 例：招聘文案模板（写好后，之后每次招聘都复用）、员工手册（最终版本）

最佳实践的流程：
Project（设计和迭代） → Artifact（最终成品） → 保存和复用
❌ 误区9：不分享 Artifact，知识无法沉淀
你自己用 AI 生成了一份很好的招聘指南、面试问题库、薪酬对标表，但只有你在用。同事还在重复做类似的工作。

问题： 无法形成团队的共识，重复劳动。
✅ 解决方案：建立团队的 Artifact 库
• 把重要的 Artifact 上传到团队 Wiki、Notion 或 GitHub
• 给同事分享 Artifact 链接或下载版本
• 定期审视和更新，形成"团队标准"
• 新员工来了，直接用团队的 Artifact 作为参考和学习资料
❌ 误区10：忽略 AI 的回答中的风险提示
AI 可能会说"这是一般建议，不构成法律意见"，但你忽略了，直接拿去用。

问题： 涉及法律、合规的事项，AI 的回答可能不准确，你承担风险。
✅ 解决方案：对于任何涉及法律、隐私、合规的问题，AI 的回答只能作为参考。一定要咨询专业人士（律师、HR 顾问）。
❌ 误区11：混淆 Skill 和 Automation（特别在 Codex 中）
有人认为 Skill 和 Automation 是替代关系，不知道何时用哪个。

问题： 两者实际上是上下层关系。误解可能导致工作流程设计不合理。

区别：
• Skill = 模板 / 操作手册。定义"怎么做"，包含说明、资源和可选脚本。可以手动调用，也可以被 AI 根据需要自动选中。
• Automation = 定时器 / 后台调度器。定义"什么时候重复做"。让 Codex 在后台按计划反复运行某个任务，结果发到 Triage/Inbox；对 Git 仓库还能选择在本地项目或独立 worktree 里跑。

对比 Claude.ai 网页版： Claude.ai 没有 Automation 功能，所有操作都是手动触发。Skill 在 Claude.ai 中也可以使用，作用相同。
✅ 解决方案：
• 先用 Skill 定义工作流程本身（如"每天的日报怎么写"、"候选人评估的标准"）
• 再用 Automation 定义执行计划（如"每天早上10点自动生成日报"、"每周五自动汇总招聘进度"）
• 官方文档提示：Automation 可以结合 Skill 使用，在自动化提示里用 @skill-name 显式触发某个 skill
• 如果你只需要一个"工作模板"，用 Skill；如果需要"定时重复执行"，用 Automation；如果需要"定时执行某个特定流程"，就同时用 Skill + Automation。
⚠️ 误区12：忽略 AI 的隐性偏见
AI 的训练数据来自互联网，其中包含很多不公平的观点和历史偏见。如果你没有显式告诉 AI 你的价值观，它可能默认延续这些偏见。

偏见来源：
• 训练数据偏见 — 互联网本身就存在性别、年龄、种族等方面的偏见
• 历史数据偏见 — 如果你上传的历史招聘数据中某类人群录用率低，AI 可能学到这个模式
• 表述偏见 — AI 学到的刻板印象（如"年轻人跳槽率高"、"女性可能因生育请假"）

后果： 招聘偏见（歧视某类候选人）、绩效评估偏见（对某些群体评分偏低）、法律风险（公司被告违反反歧视法）
✅ 解决方案：
• 在 Prompt 中声明价值观："我们遵循反歧视原则，不接受基于性别、年龄、种族的建议"
• 避免上传有偏见的历史数据
• AI 的关键建议要经多人审核，减少单一视角偏见
• 定期检查 AI 建议是否对某个群体不公平
⚠️ 误区13：不做审核，直接使用 AI 输出
拿到 AI 的输出就直接用，没有人工复核环节。尤其涉及关键业务决策时，这种做法风险极大。

正确的认知： AI 输出 = 草稿，不是最终结论。最终决策权和审核权必须在人。

不同风险等级的审核方式：
• 低风险（如邮件初稿）→ 快速过一遍，改改措辞就能用
• 中风险（如招聘分析、薪酬对标）→ 有经验的 HR 主管详细审核，验证关键数据
• 高风险（如人事决定建议、法律咨询）→ 多人复审 + 专业咨询（法务、HR 顾问）

⚖️ 责任归属： 如果因 AI 建议造成问题（如错误的招聘决定、违法的人事处理），法律责任在公司，不在 AI。审核必须严谨。
✅ 解决方案：建立分级审核机制。低风险内容快速过，中高风险内容必须经过至少一人复核。涉及法律、合规的，必须咨询专业人士。永远把 AI 当作"参谋"，不是"决策者"。

🎯 总结： AI 的最佳使用姿态是"辅助而不替代，加速而不自动，参考而不盲从"。把 AI 当作一个聪明的助手，但最终的决策权和责任权始终在人。

🔐 数据安全

1️⃣ 员工隐私保护

🚫 禁止上传真实员工信息：

  • ✗ 真实姓名、工号、身份证号
  • ✗ 电话号码、家庭住址、紧急联系人
  • ✗ 生物识别信息（照片用于识别）
  • ✗ 医疗、心理健康相关记录

✅ 信息脱敏的正确做法：

方案1：泛化处理
❌ 错误：「张三，25.5k，3年经验，北京」
✅ 正确：「P3级员工，25-30k范围，3年经验，一线城市」
方案2：编号替代
❌ 错误：把真实员工数据列成表格分析
✅ 正确：使用「员工A、员工B」或「EMP001、EMP002」
方案3：数据聚合
❌ 错误：分析单个员工的薪酬和绩效
✅ 正确：「该部门平均薪酬30k，中位数28k，80%员工在25-35k范围」

💡 实操提示： 在向AI上传数据前，问自己："如果这份数据被泄露，会对员工造成什么影响？"如果答案是"有较大风险"，就必须脱敏。

2️⃣ 敏感信息处理（薪酬、绩效评估等）

🔴 高风险信息 — 禁止上传原始数据

  • 薪酬信息： 不传「李四月薪25k」，传「P3级别市场薪酬25-35k」
  • 绩效评分： 不传「王五评分3.2分」，传「该评分等级分布：优秀10%、良好70%、需改进20%」
  • 健康/隐私： 完全禁止上传（孕期、生病、心理问题等）
  • 背景调查信息： 不传详细的背景信息，仅传「通过/未通过」

🟡 中风险信息 — 需谨慎处理

  • 离职原因分析： 可传「员工离职率、部门、职级」，但不传「个人离职原因」
  • 招聘反馈： 可传「5个候选人，2个通过，3个未通过」，不传「具体理由和评语」
  • 学历/背景： 脱敏后可用（如「985大学」而非「清华大学」）

✅ 正确的用法示例

「薪酬对标」
❌ 上传：实际员工名单 + 薪酬
✅ 上传：「公司P3级别分布：5人，薪酬范围25-30k，平均27.5k」
「绩效分析」
❌ 上传：每个员工的评分和评语
✅ 上传：「本期绩效分布：A等15%, B等70%, C等15%。主要改进方向：技术深度、团队协作」
「离职分析」
❌ 上传：「员工A因为与经理不合离职」
✅ 上传：「过去6个月，技术部离职率8%，高于公司平均4%。主要离职者职级为P3-P4」

3️⃣ 第三方AI工具的数据存储与传输策略

在使用任何AI工具前，问这5个问题：

  1. 数据存储位置： 数据存储在哪个国家/地区？是否符合数据本地化要求？
  2. 数据保留时间： AI厂商会保留我的数据多久？能否立即删除？
  3. 数据用途： 我的数据是否会被用于训练AI模型？是否会被其他用户看到？
  4. 加密和访问控制： 数据在传输和存储时如何加密？谁有权限访问？
  5. 审查和合规： 该工具是否通过了公司内部安全审查？是否有用户协议和隐私政策？

🔒 主流AI工具的数据政策参考

  • Claude (Anthropic)： 默认不保留对话内容用于模型训练；企业版可自托管
  • ChatGPT (OpenAI)： 免费版可能被用于改进模型；付费企业版有专属选项
  • 内部私有部署： 最安全，但成本和维护复杂度最高

4️⃣ 信息脱敏的操作步骤

5步脱敏流程

  1. 识别敏感字段 — 浏览数据，找出所有可识别个人的字段（姓名、工号、邮箱等）
  2. 选择脱敏方法 — 删除、替换、泛化或加密，根据数据类型选择
  3. 创建映射表 — 记录脱敏前后的对应关系（仅自己保存，不上传给AI）
  4. 验证脱敏效果 — 检查脱敏后的数据是否仍能识别个人身份
  5. 上传和使用 — 只上传脱敏后的数据给AI，保留原始数据的备份在本地

⚠️ 脱敏后要做的事： 脱敏数据使用完后，删除所有上传给AI的文件。如果数据涉及员工隐私，再做一次脱敏检查，确保没有泄露风险。

📋 合规性

1️⃣ 劳动法相关 — 自动化决策的法律风险

🚫 AI不能单独做出的决定：

  • 招聘拒绝决定 — AI可以初筛，最终拒绝需人工确认
  • 晋升/降级决策 — AI可提供数据分析，决定权在管理者
  • 绩效等级评定 — AI可提示异常值，评级权在评估者
  • 解聘决定 — 完全禁止由AI自动做出，必须经过完整的人力审查和法务确认
  • 薪酬决定 — AI可建议范围，最终核定需人工确认

✅ HR场景中AI的正确角色

  • 招聘初筛： AI帮你快速过滤不符合基本条件的简历（节省时间）→ 人工复审和面试
  • 数据分析： AI生成薪酬对标、员工留存风险分析 → 管理层参考做决策
  • 提案和建议： AI提出改进意见，如「可考虑推出远程工作政策」 → 组织讨论和评估
  • 流程自动化： AI自动生成体检提醒、入职清单 → 但涉及重要决定的仍需人工审批

⚖️ 法律风险提示

在中国，如果因为AI的建议导致员工被违法解雇、被不公平对待，公司和决策人需承担法律责任。即使AI做了推荐，人也要审核。建议：对所有涉及重要人事决定的AI应用进行法务审查。

2️⃣ 数据保护法规 — 本地化存储与访问要求

中国企业需要了解的法规

  • 《个人信息保护法 (PIPL)》： 禁止任意跨境传输个人信息。包含员工信息、应聘者信息的数据，默认不能发送到国外服务器
  • 《数据安全法》： 重要数据（如员工花名册、薪酬表）应本地存储
  • 《劳动法》第三十八条： 保障员工知情权，AI使用可能影响员工，需告知

✅ 合规操作方案

脱敏后上传（推荐）
• 在本地脱敏数据（删除敏感标识）
• 上传脱敏数据给国外AI服务
• 用脱敏后的结果，结合本地员工信息做决策
国内私有化部署
• 在公司内部服务器部署AI模型
• 所有数据不离开公司
• 成本最高，但最安全
向员工披露
• 在HR政策或员工手册中说明：「HR工作中可能使用AI辅助决策，但最终决策由人工做出」
• 如果用AI做数据分析涉及员工，建议知会员工或工会

3️⃣ 行业规范与内部审批流程

建议的内部AI治理框架

第一步：建立AI使用审批制度

  • 低风险应用（如生成邮件模板）→ HR主管批准即可
  • 中风险应用（如数据分析）→ HR总监 + IT安全审查
  • 高风险应用（如涉及人事决定）→ HR总监 + 法务 + 信息安全部门 + CEO或董事会批准

第二步：建立使用日志和记录

  • 记录：何时、何人、用AI做了什么、输出了什么、如何审核的
  • 目的：可追溯性。如果出现争议，能证明整个决策过程都经过人工审核
  • 保留期：建议至少1年

第三步：定期审查和更新规范

  • 每季度审查一次AI使用情况和合规风险
  • 如果发现问题，及时调整规范
  • 培训所有使用AI的员工了解最新规范

💡 参考模板： 可以参考《生成式人工智能服务管理暂行办法》等国家级指南，制定符合公司规模的内部政策。

✔️ 使用规范

1️⃣ 哪些场景不适合用AI？

🚫 禁止场景

  • 涉及员工隐私的敏感决定 — 例如判断某员工是否应该被解聘、是否有心理健康问题等
  • 完全自动化的人事决策 — 例如让AI完全自动审批假期申请、调薪申请等（必须有人工最后审核）
  • 上传未脱敏的员工数据 — 含真实姓名、联系方式、身份证号等的数据表
  • 直接用AI的输出作为员工沟通 — 特别是敏感沟通（辞退通知、降薪通知等）必须经过人工个性化处理
  • 用AI做出承诺性的决定 — 例如「AI说可以给你这个薪资」而没有经过正式的谈判和审批
  • 忽视法律咨询 — 涉及法律风险的HR决定，不能只依赖AI分析，必须咨询律师

✅ 适合场景

  • 信息整理和总结 — 整理面试笔记、总结员工反馈
  • 内容生成（需审核） — 写招聘文案、邮件模板、员工手册等（必须修改和审核）
  • 数据分析（脱敏数据） — 分析薪酬趋势、员工流失风险等（基于脱敏和聚合数据）
  • 流程优化建议 — AI提出改进招聘流程、绩效管理流程的想法
  • 初步筛选 — 简历初筛、应聘表初步审核（最终还需人工复审）
  • 知识库和FAQ — 用AI生成公司政策的常见问题解答

2️⃣ 信息脱敏的操作步骤（详细版）

脱敏前的检查清单

  1. 识别数据
    • □ 员工姓名 → 用「员工A」或「EMP001」替换
    • □ 工号/ID → 删除或用新ID替换
    • □ 邮箱/电话 → 删除
    • □ 身份证号 → 删除（仅保留「身份已验证」标记）
  2. 脱敏薪酬数据
    • □ 「25k」→ 「25-30k范围」或「P3级别市场薪资」
    • □ 「张三加薪8%」→ 「该级别平均加薪6-10%」
  3. 脱敏绩效数据
    • □ 「王五评分3.2/5」→ 「该评分等级人数占15%」
    • □ 不传具体评语，只传分布
  4. 脱敏背景信息
    • □ 「毕业于清华大学」→ 「顶级985大学」
    • □ 「在Google工作3年」→ 「在大型科技公司工作3年」
  5. 验证脱敏效果 — 脱敏后的数据，是否仍然能识别出具体某个人？如果能，继续脱敏

⚠️ 脱敏后的处理： 脱敏数据用完后，删除所有上传给AI的文件。原始数据保存在公司内部受保护的位置。如果涉及多个员工的数据，脱敏要特别谨慎，因为多个特征的组合可能仍然能识别个人。

✅ 使用记录的目的

• 可追溯性：出了问题能查到整个决策过程
• 合规证明：证明公司有严格的AI治理
• 持续改进：发现问题后，调整审批标准或使用方式

不同风险等级的审批要求

🟢 低风险 — 快速审批

  • 例：生成邮件模板、写招聘文案
  • 审批：HR主管批准即可，无需其他部门
  • 记录：简单记录使用日期、用途、谁在用

🟡 中风险 — 标准审批

  • 例：分析员工数据、薪酬对标、离职风险分析
  • 审批：HR总监 + IT/安全部门批准
  • 要求：数据必须脱敏；详细的使用记录；定期审查

🔴 高风险 — 严格审批

  • 例：涉及招聘/晋升/解聘的决策支持、法律意见
  • 审批：HR总监 + 法务 + CTO/CISO + CEO/董事会
  • 要求：多轮审批；详细的设计文档；第三方安全评估；建立反馈机制

🎯 AI使用的黄金法则

  1. 人为主，AI辅助 — AI只是工具，最终决策权在人
  2. 数据脱敏，隐私优先 — 任何员工个人信息上传前，先脱敏
  3. 关键决定需审批 — 涉及人事的重要决定，必须经过多人审核
  4. 结果要验证 — 数据相关的AI输出，必须自己查证
  5. 流程要记录 — 用AI的过程要有迹可查，便于后续审查
  6. 定期审视和改进 — 每季度检查一次AI的使用情况，是否合规、有效、有风险
